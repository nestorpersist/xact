Syntax
<qualified> ::= <primary>
<qualified> ::= <qualified> S? '.' S? <name> {S? '#' S? <primary>}?
<qualified> ::= <qualified> S? '(' S? {<exps> S?}? ')' {S? '#' S? <primary>}?

Description

There are two kinds of qualified expressions

  • Dot. This form is used to select a named component of the value of the expression before the dot.
  • Subscript/construct/call. This form is used to select a numbered component of the value of the expression before the parenthesis, to construct an object, or to call a function.

The semantics of qualified expression in the simple case is controlled by the type of initial expression. It is also possible to select an alternate semantics by specifying a type or view following a sharp.

Semantics

The semantics of dot qualified expression is given by by the parsing transforms specified below and the semantics of the Dot function.

The semantics of subscript/construct/call qualified expression depends upon the type of value from evaluating the initial expression.

  • Function. If the value has type func, then that function is called passing the values of the expressions inside the parenthesis as parameters.
  • Type Constructor. If the value has type type, then a new object of that type is created. If defined, the sys:Init function for that object is then called. For example, if T is a type then
       T(a,b,c)
    creates a new object, X of type T and if HasDot(T,"sys:Init") then calls
       X.sys:Init(a,b,c)
  • View Constructor. The view sys:Make function is called. For example if V is a view, then
       V(a,b,c)
    produces
       V.sys:Make(a,b,c)
    By convention the result should be in view V. So here
       V(a,b,c)~V
  • Subscript. Otherwise, the sys:Subscript function for the value is called. For example,
       X(i,j)
    results in
       X.sys:Subscript(i,j)
In all except the subscript case, the sharp and expression after the parenthesis are not permitted. For subscript, the following example
   X(i,j)#V
results in
   X.sys:Subscript#V(i,j)

Parsing

When parsed subscript/call qualified expressions are represented by xcom:call nodes.

When parsed, a dot qualified expressions are represented by xdom:call nodes whose kind is set to "dot". For example, each dot expression in the first column and the call expression in the second column are represented by identical xdom:call nodes, except the first has kind "dot" and the second has kind "normal".
a.b Dot(a,"b")
a.b#c Dot(a,"b",c)
a.b[L] Dot[L](a,"b")
a.b[L]#c Dot[L](a,"b",c)
Note that the Dot function is more general that the dot qualified expression, because the selected name can be a computed value rather than a constant.
