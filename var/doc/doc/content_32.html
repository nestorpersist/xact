The major @x:Act composite data structure is xdom trees. In addition to trees it can be used to represent directed graphs, records and arrays. The first element of an array will have index 1 (unlike Java where the first element has index 0). Arrays have a flexible size so elements can be inserted and deleted. Records are also flexible with the ability to add and delete named elements.

At least within main memory, xdom trees are really more general than just trees and can have sharing and cycles enabling xdoms to represent arbitrary directed graphs. The Unparse function in the simple case replicates for sharing and goes into an infinite loop when cycles are present. This is a special option to Unparse that can preserve sharing and cycles.

The most complex of the xdom types is xdom:element that represents xml elements and @x:Act commands. An element has properties (such as its tag), attributes and a body. A set of additional views are present for working with these various kinds of information.

  • Dot qualification on type xdom:element is used to get and set properties.
  • Array subscripting on view attr is used to get and set the ith attribute.
  • Array subscripting on view body is used to get the ith body subcomponent.
  • Dot qualification on view rec is used to get and set named attributes.
  • Array subscripting on view array is used to get and set positional attributes.
  • Dot qualification on view bodyrec is used to get body elements with the specified tag.

There are also several views that can be used to construct xdom:element values.

  • The rec view can be used to construct an element with a named set of attributes.
  • The array view can be used to construct an element with a specified set of positional attributes.
  • The list view can be used to construct an element with a list of specified body elements.

The xdom:element constructor normally takes a rec of attribute values and a list of body elements to build a new element with both attributes and body elements.
