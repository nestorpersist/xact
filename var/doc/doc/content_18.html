The result of parsing an @x:Act <script> is represented by an xdom (@x:Act Document Model) parse tree.

A <script> can be converted to an xdom using the built-in Parse function. An xdom can be converted to a <script> using the library Unparse function. Combining the Parse and Unparse functions yields a pretty printer. The pretty printer preserves code and comments but reformats an @x:Act <script> by changing white space (sequences of space and end-of-line characters). The Unparse function also provides an option to produce a color marked-up XHTML version of the script.

Having a simple xdom representation of all @x:Act <script>s makes it easy to develop @x:Act program generation, analysis, and transformation tools.

To make it simple to process xdom trees, there are only 7 types of tree nodes. The @x:Act syntax however contains lots more than 7 forms. These additional forms are considered to be "syntactic sugar" and are mapped to on of the 7 xdom types. Each xdom node contains a kind field that specifies which alternate syntax form it was parsed from. The Unparse function uses the kind field to restore the original richer syntax forms. In the xdom tree the two major forms of syntax use the same nodes but have different kinds. So conversion between forms involves simply walking the xdom tree and changing kinds.

The 7 types of xdom tree nodes are listed here together with some of the syntax forms that are mapped to these kinds of node.

  • xdom:element: <simple cmd>, <compound cmd>, <element>, <xml> comments, comments (<compound cmd>), the ?xml <element>
  • xdom:name: <name>, <escape>
  • xdom:string: <string>, <xml> text
  • xdom:int: <int>
  • xdom:float: <float>
  • xdom:call: call/subscript/dot qualification (<qualified>), operator <exp>
  • xdom:value: Allows an arbitrary @x:Act value to be store in an xdom tree. Support in Parse and Unparse is limited for this kind of node.
